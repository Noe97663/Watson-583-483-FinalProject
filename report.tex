\section{Project Overview}

A Watson-style Jeopardy QA system over a 280,000-page Wikipedia subset.
Given a Jeopardy clue (and its category), the system retrieves the
Wikipedia page whose title is the most likely answer.

The IR backend is \href{https://whoosh.readthedocs.io/}{Whoosh}, a
pure-Python search engine in the spirit of Lucene; retrieval ranking is
BM25F. The Question~5 deliverable layers a Claude Sonnet 4.6 reranker
on top of the Whoosh top-10. Final P@1 across the three pipeline modes:

\begin{itemize}
  \item \texttt{baseline} (Whoosh BM25F): \textbf{0.20}
  \item \texttt{improved} (phrase + capitalized-run boosts + rerank):
        \textbf{0.25}
  \item \texttt{llm} (improved + Claude Sonnet 4.6 reranker over the
        top-10): \textbf{0.55}
\end{itemize}

The LLM mode achieves $\text{P@1} = \text{P@5} = \text{P@10} = 0.55$:
the reranker correctly identifies the gold page on every clue where
retrieval surfaced it in the top-10. The remaining 45 errors are pure
retrieval failures, bucketed in Section~\ref{sec:error}.

\section{How to Compile and Run}

\subsection{Installation}

No compilation step is required. Install runtime dependencies:

\begin{lstlisting}[language=bash]
python -m pip install -r requirements.txt
\end{lstlisting}

\texttt{requirements.txt} pins two packages: \texttt{whoosh==2.7.4} for the
IR layer and \texttt{anthropic>=0.45.0} for the Claude reranker used in
mode \texttt{llm} (see Question~5). Tests use Python's built-in
\texttt{unittest}; only the LLM-mode runs require an API key.

\subsection{Build the Index (one-time, \textasciitilde30--60 min)}

\begin{lstlisting}[language=bash]
python src/build_index.py --data-dir data --index-dir index
\end{lstlisting}

This streams every wiki file in \texttt{data/}, parses out individual pages,
strips \texttt{[tpl]\ldots[/tpl]} and \texttt{[ref]\ldots[/ref]} markup, drops
\texttt{\#REDIRECT} stubs, and indexes the rest with a Porter-stemming and
stopwords analyzer. Each Wikipedia page becomes one document (not one document
per file).

\subsection{Run the Evaluation}

Both pure-IR modes side-by-side:

\begin{lstlisting}[language=bash]
python src/compare_modes.py --index-dir index --questions data/questions.txt --out-dir results
\end{lstlisting}

All three modes (including the Claude reranker --- needs an Anthropic API
key, see below):

\begin{lstlisting}[language=bash]
python src/compare_modes.py --modes baseline improved llm
\end{lstlisting}

A single mode (with per-question table printed to stdout):

\begin{lstlisting}[language=bash]
python src/evaluate.py --mode baseline --out results/baseline.json
python src/evaluate.py --mode improved --out results/improved.json
python src/evaluate.py --mode llm      --out results/llm.json
\end{lstlisting}

\textbf{For the \texttt{llm} mode}, the API key can be supplied either via
the standard \texttt{ANTHROPIC\_API\_KEY} environment variable or by writing
it to \texttt{.env} at the project root as \texttt{api\_key=sk-ant-\ldots}.
The reranker caches every decision to
\texttt{results/llm\_rerank\_cache.json}, so re-runs cost nothing.

\subsection{Error Analysis}

\texttt{error\_analysis.py} bins each miss into one of five heuristic
buckets and prints both a count summary and per-bucket detail:

\begin{lstlisting}[language=bash]
python src/error_analysis.py results/llm.json --summary-only
python src/error_analysis.py results/llm.json --show pun_category
\end{lstlisting}

\subsection{Tests}

\begin{lstlisting}[language=bash]
python -m unittest discover -s tests
\end{lstlisting}

\section{Question 1 --- Core Implementation: Indexing and Retrieval (50 pts)}

\subsection{Engine}

We used Whoosh, a pure-Python IR library with BM25F default ranking.
Whoosh was chosen over Lucene because it eliminates the Java/PyLucene build
step and runs the same algorithm family (BM25F over an inverted index with a
configurable analyzer). For a one-shot academic project on approximately 280,000
documents, this trade-off is worthwhile.

\subsection{Indexing Pipeline (\texttt{wiki\_parser.py}, \texttt{build\_index.py})}

Each wiki text file is streamed line by line: \texttt{wiki\_parser.parse\_pages}
opens one file at a time and yields \texttt{(title, body)} tuples without ever
loading more than a single page into memory. A page boundary is a line of the
form \texttt{[[Title]]} (the entire line) that does not start with a wiki
namespace prefix (\texttt{File:}, \texttt{Image:}, \texttt{Category:},
\texttt{Wikipedia:}, \texttt{Template:}). This same regex would otherwise match
in-line image embeds like \texttt{[[File:Foo.svg|thumb|caption\ldots]]}, so we
exclude those by both prefix and by requiring \texttt{]]} to close on the same
line (image embeds frequently span many lines).

\subsection{Term Preparation}

Whoosh's \texttt{StemmingAnalyzer} is applied to both fields. It performs:

\begin{itemize}
  \item Lowercasing
  \item Standard English stopword removal (\emph{the}, \emph{a}, \emph{of}, \ldots)
  \item Porter stemming (so \emph{paper / papers / papered} collapse to one token)
\end{itemize}

\subsection{Schema}

\begin{table}[h]
\centering
\begin{tabular}{lll}
\toprule
\textbf{Field} & \textbf{Type} & \textbf{Stored?} \\
\midrule
\texttt{title}     & \texttt{TEXT(StemmingAnalyzer, boost=2.0)} & yes \\
\texttt{body}      & \texttt{TEXT(StemmingAnalyzer)}            & no  \\
\texttt{raw\_title} & \texttt{STORED}                            & yes \\
\texttt{doc\_id}   & \texttt{ID(unique=True)}                   & yes \\
\bottomrule
\end{tabular}
\caption{Whoosh index schema}
\end{table}

The \texttt{title} field is boosted ($\times$2) so that a clue that mentions
the actual page title receives a strong signal. \texttt{raw\_title} is kept
verbatim for answer comparison (the analyzed title would be lowercased and
stemmed).

\subsection{Wiki-Specific Issues Encountered}

\begin{enumerate}
  \item \textbf{Redirects.} Approximately 10\% of pages are
        \texttt{\#REDIRECT Foo} stubs with no body of their own. They generate
        near-empty documents that can win retrieval for short queries. They are
        detected and skipped at parse time.

  \item \textbf{Citation/reference noise.}
        \texttt{[tpl]cite web|url=\ldots|publisher=\ldots[/tpl]} blobs and
        \texttt{[ref]\ldots[/ref]} tags add hundreds of irrelevant tokens
        (publisher names, URLs, dates) that drift the BM25 score. They are
        stripped before indexing.

  \item \textbf{Image/file embeds masquerading as page titles.} Lines like
        \texttt{[[File:Vegetation-no-legend.PNG|thumb|center|800px|\ldots]]}
        are filtered by namespace prefix and by requiring \texttt{]]} on the
        same line.

  \item \textbf{CATEGORIES line.} Each page's \texttt{CATEGORIES: a, b, c}
        line is a strong topical signal. We keep the category names but strip
        the literal \texttt{CATEGORIES:} label so it does not itself get indexed
        as a content word.

  \item \textbf{Section-header equals signs and link bracketing}
        (\texttt{==History==}, \texttt{[[A|B]]}). Headers are kept as plain
        text; piped links are reduced to their display string; bare
        \texttt{[[X]]} links are reduced to their target.

  \item \textbf{Duplicate page titles across files.} Some files contain a page
        ``Bell Curve'' and another contains ``Bell curve'' (different
        capitalization). The unique \texttt{doc\_id} is de-duplicated by
        appending \texttt{\#2}, \texttt{\#3}, etc.\ so neither is silently
        overwritten in the index.
\end{enumerate}

\subsection{Query Construction (\texttt{watson.py})}

The clue is sanitized (Whoosh-syntax characters \texttt{:}, \texttt{\^{}},
\texttt{(}, \texttt{)}, \texttt{*}, \texttt{?} are replaced by spaces) and
parsed by \texttt{MultifieldParser} against \texttt{title} and \texttt{body}
with \texttt{OrGroup} semantics. Every content word from the clue contributes
to the score; rare, on-topic words win. The system does \emph{not} select a
subset of clue words --- empirically that performs worse than letting BM25
plus stopword removal handle the down-weighting.

Several variants were explored:

\begin{table}[h]
\centering
\begin{tabular}{lr}
\toprule
\textbf{Variant} & \textbf{P@1} \\
\midrule
Clue only (baseline)                                         & 0.20 \\
Clue + category as raw extra terms\textsuperscript{*}        & 0.12 \\
Clue + $\times$2 boost on every mid-clue capitalized word\textsuperscript{*}    & 0.15 \\
Clue + phrase boost + capitalized-run boost + soft rerank (improved) & 0.25 \\
\bottomrule
\end{tabular}
\caption{P@1 for different query construction strategies.
\textsuperscript{*}\,Numbers from earlier prototype runs that were
ablated into the final \texttt{improved} mode --- the corresponding
code paths are no longer in \texttt{watson.py}, so they cannot be
recomputed from the current source tree without temporarily reverting
\texttt{\_build\_query\_string}. The \texttt{baseline} and
\texttt{improved} rows are reproducible from the shipped code.}
\end{table}

The improved mode makes three deliberate choices:

\begin{itemize}
  \item \textbf{Quoted phrases} (e.g.\ \emph{``Adoration of the Golden Calf''},
        \emph{``Good Will Hunting''}) are extracted and added as \texttt{\^{}3}
        phrase queries. These are almost always the disambiguating named
        entities.
  \item \textbf{Multi-word capitalized runs} (e.g.\ \emph{Pierre Cauchon},
        \emph{El Tahrir}) are added as \texttt{\^{}2} phrase queries. Single
        capitalized words are deliberately \emph{not} boosted --- boosting
        \emph{Hunting} and \emph{Oscar} in the Ben Affleck clue pulled
        retrieval toward \emph{Helen Hunt} and \emph{Oscar Wilde}.
  \item \textbf{Category is intentionally dropped} from the BM25 query.
        Short, hyped category strings like \texttt{1920s NEWS FLASH!} drown
        out the actual clue (an early prototype regressed \emph{Ottoman Empire}
        to off-list exactly this way).
\end{itemize}

\section{Question 2 --- Coding with LLMs (50 pts)}

\subsection{Which LLMs and How Prompted}

Two Claude variants were used in this project:

\begin{itemize}
  \item \textbf{Claude Code (Opus 4.7, 1M context)} as the development
        assistant in an interactive terminal session. All source code,
        tests, README, and this report were authored or substantially
        edited with Claude Code. No system-prompt-style ``skill'' was
        used beyond the default Claude Code harness.
  \item \textbf{Claude Sonnet 4.6} via the Anthropic Python SDK as the
        runtime reranker for Question~5 (\texttt{src/llm\_rerank.py}).
        Sonnet was chosen over Opus 4.7 here because reranking 10
        candidate Wikipedia titles is constrained classification, not
        open-ended reasoning, and Sonnet is approximately $3\times$
        cheaper for the same accuracy on this workload.
\end{itemize}

\paragraph{The \texttt{instructions.md} bootstrap.} The project's own
\texttt{instructions.md} file --- the natural-language brief that drove
every later coding session --- was itself first generated by an LLM
from \texttt{project.md}. The LLM was asked to read the spec, identify
the discrete deliverables, and write a minimal-friction set of
instructions another LLM (or human) could follow without re-reading
the entire spec. That generated file became the entry point for the
development process: subsequent Claude Code sessions started with
``Follow \texttt{instructions.md}'' rather than ``Read
\texttt{project.md} and figure out what to do.'' This eliminated a
round of spec re-parsing on every session resume.

Subsequent prompts to Claude Code were natural-language and short
(``Follow the instructions in \texttt{instructions.md} to complete
the project.'', ``Review changed code for reuse and quality'',
``Bucket the errors and update the README''), with the model working
autonomously --- reading the spec, sampling the data, proposing an
architecture, and producing the pipeline.

\subsection{How the LLM Was Used}

The human submitter supplied the project spec and the data, and acted as
orchestrator. The LLM:

\begin{enumerate}
  \item Read \texttt{instructions.md} and \texttt{project.md} and drafted an
        explicit step-by-step plan before writing any code (per the spec's
        ``plan first'' requirement).
  \item Sampled \texttt{questions.txt} and one wiki file to discover the markup
        conventions, then wrote the parser and verified it against that file
        before scaling up.
  \item Wrote the index builder, the retrieval module, the evaluator, and a set
        of \texttt{unittest} tests.
  \item Built the index, executed the evaluation, performed the error analysis,
        and wrote the README.
\end{enumerate}

This was closer to \textbf{vibe-coding with active oversight} than either pure
vibe-coding or ``LLM as Stack Overflow.'' The architecture
(parser $\to$ index $\to$ retrieval $\to$ evaluator $\to$ error analysis) was
arrived at by the LLM after reading the spec; each module was reviewed and
approved before the next one started, but the design proposals were not
human-supplied.

\subsection{What Worked}

\begin{itemize}
  \item The ``stream wiki files; one document per page'' idea was inferred
        directly from the spec's \texttt{[[Title]]} description without
        hand-holding the parser logic.
  \item Discovering the gotchas (redirect stubs, multi-line
        \texttt{[[File:\ldots{}} image embeds, \texttt{CATEGORIES:} line) came
        from the LLM actually reading a real wiki file and noticing them, not
        from priors.
  \item BM25F + stemming + title boost is exactly the off-the-shelf default one
        would pick --- the LLM did not over-engineer.
  \item Tests caught a real bug: the first version of
        \texttt{\_build\_query\_string} unconditionally appended the category,
        which the test \texttt{test\_baseline\_query\_omits\_category\_when\_none}
        flagged immediately.
\end{itemize}

\subsection{What Did Not Work}

\begin{itemize}
  \item The first improved-mode reranking idea was too aggressive: dropping all
        pages whose title shared \emph{any} token with the clue threw out
        legitimate answers. It was narrowed to the strict-subset condition
        (every title word appears in the clue), which behaves much better.
  \item The Whoosh \texttt{QueryParser} defaults to \texttt{AndGroup}, which
        produced almost no hits for long clues (every conjunct must match).
        Switching to \texttt{OrGroup} was an explicit fix.
  \item Punctuation in clues (\texttt{o'hare}, \texttt{we'll}, \texttt{Mr.})
        trips the parser if not pre-sanitized. The fix is the
        \texttt{\_QUERY\_BAD} regex; without it Whoosh raises
        \texttt{QueryParserError} on roughly a dozen of the 100 questions.
\end{itemize}

In short: the LLM produced a working system in one session, but the
\textbf{judgment calls} (which IR algorithm, which stopword list to remove,
how aggressive to make the reranker) were made better when reviewed and pushed
back on rather than accepted blindly.

\subsection{Verbatim Prompts}

To make the LLM-collaboration claims concrete, here are the two
prompts that did the most work in this project.

\paragraph{The reranker's system prompt} (verbatim from
\texttt{src/llm\_rerank.py}, used on every \texttt{llm}-mode request):

\begin{lstlisting}
You are a Jeopardy expert. Given a Jeopardy clue, its category, and a
ranked list of candidate Wikipedia page titles produced by a lexical
search engine, pick the candidate whose page is the actual answer.

Notes on the task:
- The clue is a statement; the answer is the topic the statement
  describes (a person, place, work, event, or concept whose name is
  the title of a Wikipedia page).
- Categories often contain wordplay or puns and can be a strong hint.
  For example, a category like "TIN" MEN means the answer's name
  contains the substring "tin" -- Tintoretto, Quentin Tarantino, etc.
- The lexical retriever can miss the right answer when the connection
  is INFERENTIAL rather than literal. Examples:
    * "Rebel with a cause" -> James Dean (via Rebel Without a Cause)
    * Clue mentions a museum -> answer is the city or state
    * A pun in the category encodes the disambiguator
  Use your knowledge to bridge those gaps.
- Conversely, the right answer is sometimes obvious from the clue
  alone and the lexical retriever already ranked it #1 -- in that case
  just keep it.
- The retrieved list can include "List of ...", "Index of ...", or
  disambiguation pages. Those are almost never the answer.
- If none of the candidates is plausibly correct, return
  best_index = 0. Otherwise pick the single best one -- do not refuse.

Respond with JSON: {"best_index": <int>, "reasoning": <one sentence>}.
\end{lstlisting}

The user message on each call is short:
\texttt{CATEGORY: <cat>}, \texttt{CLUE: <clue>}, then a numbered list
of the 10 candidate titles. Output is constrained to a JSON schema
via the SDK's \texttt{output\_config.format} so parsing is trivial.
Adaptive thinking is enabled; effort is set to \texttt{medium}.

\paragraph{The regression-diagnosis prompt} (paraphrased from the
session log; the actual prompt was a one-line ask plus the dumped
contingency between the two modes):

\begin{lstlisting}
Diagnose the regression: improved mode is at P@1=0.15 vs baseline
P@1=0.20. Here are the 8 questions baseline got right that improved
got wrong, with the top-5 hits each mode returned:

  Tintoretto (cat="TIN" MEN, clue=adoration of the golden calf)
    baseline top: Tintoretto, Leonardo da Vinci, ...
    improved top: Calf of Man, Master of Khanenko Adoration, ...
  Ben Affleck (cat=HE PLAYED A GUY NAMED JACK RYAN IN..., clue=
    "The Sum of All Fears", screenwriting Oscar for "Good Will Hunting")
    baseline top: Ben Affleck, John Huston, Helen Hunt, ...
    improved top: Helen Hunt, Zero-sum game, Oscar Levant, ...
  [...6 more]

What changed between baseline and improved that explains these losses?
Suggest a fix; do not write code yet.
\end{lstlisting}

Claude's response identified two failure modes within one turn:
appending the raw category string (\texttt{1920s NEWS FLASH!})
overwhelmed BM25, and per-word \texttt{\^{}2} boosts on every
mid-clue capitalized word promoted distractors like \emph{Hunting}
and \emph{Oscar} above the actual entity. The fix --- restrict
boosts to quoted phrases and multi-word capitalized runs, and drop
the category from the query entirely --- recovered the regression
and added $+5$ P@1 over baseline.

\subsection{Building the LLM Reranker (Question 5)}

The Question~5 deliverable is itself an LLM-assisted addition built in
a follow-up session. Claude:

\begin{enumerate}
  \item Inspected the \texttt{build\_index.py} schema and noticed that
        the \texttt{body} field is not stored, ruling out
        snippet-augmented prompts without a 20-minute reindex.
  \item Proposed a title-only prompt grounded in the observation that
        Claude has Wikipedia in its training data; verified the
        approach worked on a single dry-run question before committing
        to all 100.
  \item Wrote the \texttt{LLMReranker} class with a SHA-256-keyed disk
        cache, structured-output JSON via \texttt{output\_config.format},
        adaptive thinking, a \texttt{.env} loader for the API key,
        and 8 mocked-client unit tests so CI does not require a live
        API key.
  \item Recommended Sonnet 4.6 over Opus 4.7 for cost reasons (the
        reranking task is constrained classification, not open-ended
        reasoning; Sonnet is $\sim$$3\times$ cheaper and equally
        accurate on this workload).
\end{enumerate}

The pattern of the second session was less ``LLM writes code'' and
more ``LLM proposes design tradeoffs and the human chooses.'' Each
non-trivial decision (title-only vs.\ snippet, Sonnet vs.\ Opus, where
to place the disk cache) was offered with an explicit rationale; the
human accepted or redirected. This is the most productive shape of
LLM collaboration for an academic project: the LLM eliminates
implementation friction, but the human retains the architectural
choices.

\section{Question 3 --- Performance Measurement (50 pts)}

\subsection{Choice of Metric}

Both \textbf{P@1} (precision-at-1, the fraction of clues whose top retrieved
page is the gold answer) and \textbf{MRR} (mean reciprocal rank over the
top-10 hits) are reported. Other metrics discussed in class are less
appropriate here:

\begin{itemize}
  \item \textbf{Recall} and \textbf{F1} are ill-defined: a Jeopardy clue has
        exactly one correct page (modulo aliases), so recall over a top-$K$
        cutoff collapses to a hit/no-hit indicator and reduces to \textbf{P@K}.
  \item \textbf{NDCG} requires graded relevance judgments; here the labels are
        binary (this page is the answer, or it is not), so NDCG with binary
        gain is mathematically equivalent (up to a constant) to a
        rank-discounted hit rate --- i.e.\ MRR.
\end{itemize}

P@1 directly measures whether the system would \emph{answer the question} on
Jeopardy (one buzzer, one guess). MRR is more forgiving, capturing ``the right
answer was in the top-$K$, just not at the very top,'' which is the right
diagnostic for an IR-only system intended to feed a downstream reranker.

\subsection{Results on the 100 Supplied Questions}

\begin{table}[h]
\centering
\begin{tabular}{lrrrr}
\toprule
\textbf{Mode} & \textbf{P@1} & \textbf{P@5} & \textbf{P@10} & \textbf{MRR} \\
\midrule
baseline             & 0.20 & 0.42 & 0.53 & 0.303 \\
improved             & 0.25 & 0.46 & 0.55 & 0.344 \\
\textbf{llm} & \textbf{0.55} & \textbf{0.55} & \textbf{0.55} & \textbf{0.550} \\
\bottomrule
\end{tabular}
\caption{Evaluation results (top-10 retrieval window). Reproduced verbatim
from \texttt{results/baseline.json}, \texttt{results/improved.json}, and
\texttt{results/llm.json}.}
\end{table}

A striking property of the \texttt{llm} row is that
$\text{P@1} = \text{P@5} = \text{P@10} = \text{MRR} = 0.55$. The reranker
moves whichever candidate it picks to rank~1 and leaves the rest in
their original order, so identical numbers across cutoffs mean the
reranker pulled \emph{every} correct answer present in the IR top-10 to
rank~1. The remaining 45 errors are pure retrieval failures: the gold
page never made it into the top-10 in the first place. The ceiling is
now retriever recall, not ranking quality.

\subsection{Comparison Detail}

Of the 100 questions:

\begin{itemize}
  \item 12 questions that baseline answered correctly are also correct
        under improved (no churn).
  \item 8 questions improved gets right that baseline did not
        (quoted-phrase and capitalized-run boosts win, e.g.\
        \emph{George Martin}, \emph{Joe Tinker}, \emph{Michael Jackson},
        \emph{Janet Jackson}, \emph{Heather Locklear}).
  \item 3 questions baseline got that improved loses (\emph{Tintoretto},
        \emph{Game Change}, \emph{Ottoman Empire}) --- see
        Section~\ref{sec:error}.
\end{itemize}

Net for the rule-based improvements: \textbf{+5 P@1, +0.04 MRR}.

For the \texttt{llm} mode, the per-question diff against baseline is
\textbf{+37 gains and 1 loss} (the 1 loss is \emph{Ottoman Empire}, which
falls out of the top-10 entirely in retrieval, so the reranker has no
candidate to pick from). Two of the three improved-mode regressions
(\emph{Tintoretto}, \emph{Game Change}) are recovered by the LLM
because the gold page is still in the top-10 for those.

\subsection{BM25F \texorpdfstring{$B$}{B} Parameter Sweep}

Whoosh's BM25F default $B = 0.75$ over-penalizes long Wikipedia articles (a
one-paragraph stub whose title contains a clue word can outrank a 5,000-word
biography). A sweep over the full 100-question dev set was performed:

\begin{table}[h]
\centering
\begin{tabular}{rrr}
\toprule
$B$ & P@1 & MRR \\
\midrule
0.00 & 0.15 & 0.260 \\
0.10 & 0.20 & 0.303 \\
0.20 & 0.20 & 0.297 \\
0.50 & 0.15 & 0.251 \\
0.75 & 0.12 & 0.196 \\
\bottomrule
\end{tabular}
\caption{BM25F length-normalization parameter sweep}
\end{table}

$B = 0.1$ is baked into \texttt{watson.py} and used by both modes.
Note that this sweep was performed on the same 100-question set the
report's headline metrics are computed against --- the project spec
provides no held-out split, and 100 questions is too small to carve
one out meaningfully without losing statistical power. The
$B = 0.1$ choice is therefore mildly tuned to the eval set; a
genuinely held-out test would be the right way to confirm the choice
generalizes.

\subsection{Statistical Significance (McNemar's Exact Test)}

Because the three modes are scored on the \emph{same} 100 questions,
the right significance test for any pairwise comparison is McNemar's
test on the contingency table of paired correctness. We computed it
exactly (binomial test on the discordant cells) via
\texttt{src/significance.py}; the results:

\begin{table}[h]
\centering
\begin{tabular}{lrrl}
\toprule
\textbf{Comparison} & \textbf{Discordant pairs} & \textbf{Exact p-value} & \textbf{Verdict} \\
\midrule
baseline vs.\ improved & 11 (8 vs.\ 3) & 0.227    & not significant \\
improved vs.\ llm      & 30 (30 vs.\ 0) & $1.9 \times 10^{-9}$ & highly significant ($p<0.001$) \\
baseline vs.\ llm      & 37 (36 vs.\ 1) & $5.5 \times 10^{-10}$ & highly significant ($p<0.001$) \\
\bottomrule
\end{tabular}
\caption{McNemar exact-binomial paired tests on per-question
correctness. ``$a$ vs.\ $b$'' counts are formatted as
``$b$-only-correct vs.\ $a$-only-correct.''}
\end{table}

The \texttt{baseline vs.\ improved} gain ($+5$ P@1) is \emph{not}
significant at $\alpha = 0.05$: 8 questions improved by the rule-based
changes versus 3 that regressed, and at $n = 100$ that 8/3 split could
plausibly happen by chance ($p = 0.227$). The honest reading is that
the rule-based improvements deliver a real but small gain that this
small evaluation set cannot distinguish from sampling noise. The
$+30$ P@1 LLM-rerank gain is unambiguously real
($p \approx 2 \times 10^{-9}$), as is the cumulative
baseline-to-llm jump ($p \approx 5 \times 10^{-10}$). With only $n = 100$
questions, a single bucket shift is roughly $\pm 3$ P@1; improvements
smaller than that should be treated as noise without significance
testing.

\section{Question 4 --- Error Analysis (50 pts)}
\label{sec:error}

\texttt{src/error\_analysis.py} reads any \texttt{evaluate.py --out} JSON
dump, classifies each miss into one of five buckets via simple
heuristics, and prints both a count summary and per-bucket dumps.

\subsection{Two Perspectives on Errors}

The pure-IR \texttt{improved} mode misses 75/100 questions (25 right at
rank~1, 30 more in the top-10, 45 missed entirely). The \texttt{llm}
mode pulls every top-10 hit to rank~1 --- its 45 misses are a strict
subset of \texttt{improved}'s, namely the ones the retriever never
surfaced. \textbf{The bucket counts below are computed on the LLM mode's 45
misses}, since those are the residual errors after rule-based
\emph{and} LLM improvements have done what they can.

\subsection{Bucket Counts (45 misses / 100 questions)}

\begin{table}[h]
\centering
\begin{tabular}{lrr}
\toprule
\textbf{Bucket} & \textbf{Count} & \textbf{\% of misses} \\
\midrule
\texttt{indirect\_lookup}   & 20 & 44.4\% \\
\texttt{pun\_category}      &  9 & 20.0\% \\
\texttt{under\_anchored}    &  4 &  8.9\% \\
\texttt{decoy\_overpowered} &  2 &  4.4\% \\
\texttt{other}              & 10 & 22.2\% \\
\bottomrule
\end{tabular}
\caption{Auto-bucketed error counts on the \texttt{llm}-mode misses.}
\end{table}

\subsection{What Each Bucket Means and What We Observed}

\begin{enumerate}
  \item \textbf{\texttt{indirect\_lookup} (20 / 44\%) --- by far the largest class.}
        The \emph{category} constrains the answer to a \emph{different
        entity than the clue describes}. Retrieval goes hunting for the
        entity in the clue and gets nowhere near the answer. Sub-patterns:
        \begin{itemize}
          \item \texttt{STATE OF THE ART MUSEUM} --- clue gives the museum,
                answer is the U.S.\ state (5 instances: \emph{Florida},
                \emph{Ohio}, \emph{New Mexico}, \emph{Michigan}, \emph{Idaho}).
          \item \texttt{CAPITAL CITY CHURCHES} --- clue gives the church,
                answer is the capital (\emph{Helsinki}, \emph{Edinburgh}).
          \item \texttt{NAME THE PARENT COMPANY} --- clue gives the brand,
                answer is the parent (\emph{Kraft Foods} for Jell-O,
                \emph{3M} for Post-it).
          \item \texttt{`80s NO.1 HITMAKERS} --- clue gives the song,
                answer is the artist (\emph{George Michael},
                \emph{Michael Jackson} $\times$2).
          \item \texttt{GOLDEN GLOBE WINNERS} --- clue gives the role,
                answer is the actor (\emph{Heath Ledger},
                \emph{Kelsey Grammer}, \emph{Martin Sheen}).
          \item \texttt{1920s NEWS FLASH!} and \texttt{HISTORICAL QUOTES}
                --- clue gives the event/quote, answer is the historical figure.
        \end{itemize}
        These are unsolvable by lexical retrieval \emph{or} by a
        title-only reranker reading from the IR top-10, because the IR
        system was never going to surface the answer-side entity in the
        first place (the museum article rarely contains the clue text,
        and nothing about the clue points to the museum's state).

  \item \textbf{\texttt{pun\_category} (9 / 20\%) --- wordplay constraints in
        the category.} The category itself is a puzzle whose solution
        constrains the answer (\texttt{"TIN" MEN} $\to$ answer's name
        contains \emph{tin}; \texttt{COMPLETE DOM-INATION} $\to$ answer's
        name contains \emph{dom}; \texttt{OLD YEAR'S RESOLUTIONS} $\to$
        answer is some historical \emph{resolution}). The clue text alone is
        generally too generic to localize the answer. \emph{Tintoretto}
        (the previous improved-mode regression) was rescued by the LLM
        reranker because it appeared in the top-10; \emph{Vladimir Putin}
        under \texttt{"TIN" MEN} was not.

  \item \textbf{\texttt{under\_anchored} (4 / 9\%) --- clues with no high-IDF
        anchors.} Quoted song lyrics or 1--3 word product clues:
        \emph{``Father Figure''}, \emph{``Beat It''}, \emph{``Rock With You''},
        \emph{Jell-O}. BM25 has nothing rare to lock onto.

  \item \textbf{\texttt{decoy\_overpowered} (2 / 4\%) --- title decoys.} The
        top-1 hit is a different entity that shares more clue-content
        tokens with the clue than the gold's title does (\emph{USS Maine}
        clue $\to$ \emph{USS Arizona Memorial} hit; \texttt{WWF\ldots\ bald
        eagle\ldots\ red wolf} $\to$ \emph{Red panda}). The improved-mode
        subset-decoy demotion already catches the easiest of these; what
        remains here are cases where the decoy only partially overlaps
        with the clue.

  \item \textbf{\texttt{other} (10 / 22\%) --- body-text topical drift +
        inferential.} Title-overlap heuristics cannot see this, but
        inspection shows two sub-patterns dominate:
        \begin{itemize}
          \item \emph{Topical drift}: the IR system retrieves topically
                related pages whose \emph{body} matches the clue better
                than the answer page's body does (\emph{Wolong\ldots\
                bamboo\ldots\ China} $\to$ \emph{Chengdu}; clue about
                Wordsworth's pseudonym $\to$ \emph{Axios (acclamation)};
                clue about JFK's 1960 campaign $\to$ \emph{Richard Nixon},
                his opponent).
          \item \emph{Inferential / multi-hop}: tuk-tuk $\to$
                \emph{Rickshaw}, Ammonites $\to$ \emph{Jordan} (via Amman),
                Norodom statue $\to$ \emph{France} (via the French
                protectorate over Cambodia). Pure lexical retrieval
                cannot bridge a one-hop world-knowledge step.
        \end{itemize}
\end{enumerate}

\subsection{Implications for What to Fix Next}

\begin{itemize}
  \item The two largest buckets (\texttt{indirect\_lookup} +
        \texttt{pun\_category} = \textbf{64\%} of remaining errors)
        cannot be solved by improving the IR ranker alone. They need
        either (a) an \emph{LLM at the query side} generating expanded
        queries from \texttt{(clue, category)}, or (b) feeding the
        clue to an LLM with web/Wikipedia tool access.
  \item The next-best lever is reducing \emph{topical body drift} in the
        \texttt{other} bucket ($\sim$22\%) --- a body-side rerank that
        re-scores top-50 hits with proximity / phrase windows / first-paragraph
        match could plausibly recover several.
  \item The \texttt{decoy\_overpowered} bucket is small (2 cases) ---
        the improved-mode subset rerank already handled the obvious ones.
\end{itemize}

\subsection{Why the Easy 25 Work --- Clue-IDF Analysis}

A natural hypothesis is that the clues the baseline gets right are the
ones whose content tokens are individually \emph{rare} (high IDF), so
that BM25 has at least one strong, distinguishing anchor pointing at
the answer page. To test this quantitatively,
\texttt{src/idf\_analysis.py} computes for every clue the maximum IDF
of its content tokens against the index's body postings, then bins
the 100 questions into IDF quintiles and reports baseline P@1 per bin.

\begin{figure}[h]
\centering
\includegraphics[width=0.85\textwidth]{results/idf_histogram.png}
\caption{Baseline P@1 vs.\ the maximum content-token IDF of the clue
(equal-frequency quintiles, $n = 99$ after dropping one clue with no
content tokens). The label under each bar is the IDF range; ``n''
above each bar is the number of questions in the bin.}
\label{fig:idf}
\end{figure}

The pattern is non-monotone --- a \textbf{``sweet spot''} rather than
the simple ``rare = easy'' story we expected:

\begin{table}[h]
\centering
\begin{tabular}{lrrr}
\toprule
\textbf{Max-IDF range} & \textbf{n} & \textbf{P@1} & \textbf{Interpretation} \\
\midrule
$[1.9, 5.0]$  & 20 & 0.10 & no rare anchor in the clue        \\
$[5.1, 6.8]$  & 20 & 0.20 & moderate anchors                  \\
$[7.1, 9.8]$  & 20 & \textbf{0.45} & sweet spot          \\
$[9.9, 11.2]$ & 20 & 0.25 & very rare tokens, mixed signal    \\
$[11.2, 11.9]$ & 19 & 0.00 & ultra-rare tokens, often hapaxes  \\
\bottomrule
\end{tabular}
\caption{Baseline P@1 by clue max-token IDF quintile.}
\end{table}

Three observations:

\begin{itemize}
  \item \textbf{Clues with no rare anchor (lowest quintile) score
        $0.10$.} These are the \texttt{under\_anchored} bucket from the
        miss analysis above --- mostly quoted song lyrics
        (\emph{``Beat It''}, \emph{``Father Figure''}) or 1--3 word
        product clues (\emph{Jell-O}, \emph{Post-it notes}) where the
        clue's most distinctive token is itself common.
  \item \textbf{Sweet spot $[7.1, 9.8]$ scores $0.45$.} A single
        moderately-rare distinguishing token --- a place name,
        organization, or domain-specific noun that appears in dozens to
        a few hundred Wikipedia pages --- gives BM25 enough signal to
        lock onto the right one. \emph{Indonesia, lizard, poachers}
        for \emph{Komodo dragon} is a textbook case.
  \item \textbf{Ultra-high IDF (top quintile) scores $0.00$.} Tokens
        this rare are usually obscure proper nouns, foreign-language
        phrases, or even typos that appear in only a handful of
        Wikipedia pages --- and crucially, often \emph{not} in the page
        that's actually the answer. Examples include \emph{Tuomiokirkko}
        in the Helsinki cathedral clue (the Finnish word appears in a
        few cathedral pages but not as a strong signal toward
        \emph{Helsinki}) and \emph{Wolong} in the Panda clue (matches
        nature-preserve pages, not \emph{Panda}).
\end{itemize}

The takeaway is that high IDF is a necessary but not sufficient
condition for retrieval to work --- the rare token needs to actually
appear in the answer page's body. This refines the original ``rare =
easy'' claim into something more useful: BM25 wants \emph{discriminating}
tokens, not just \emph{rare} ones, and these are not the same thing.

\section{Question 5 --- Improved Implementation (Grad Students Only / Extra Credit) (50 pts)}

Two improvements were stacked on top of the baseline IR system: a
rule-based query-and-rerank pass (\texttt{--mode improved}) and an
LLM reranker over the top-10 (\texttt{--mode llm}). The improved mode
gains \textbf{+5 P@1} over baseline; layering the LLM reranker on
top adds another \textbf{+30 P@1}, for a final P@1 of $0.55$.

\subsection{Mode \texttt{improved} --- Rule-Based Query and Rerank}

\texttt{watson.py --mode improved} differs from the baseline in three ways:

\begin{enumerate}
  \item \textbf{Quoted-phrase boost (\texttt{\^{}3}).} Phrases inside double
        quotes in the clue (straight or curly) are extracted and added as
        phrase queries. For \emph{``Daniel Hertzberg \& James B.\ Stewart of
        this paper share a Pulitzer for their stories on insider trading''} no
        phrase fires; for \emph{``He won an Oscar for `Good Will Hunting'''} the
        phrase \emph{Good Will Hunting} is added as a \texttt{\^{}3} phrase,
        anchoring retrieval on the named work.

  \item \textbf{Multi-word capitalized-run boost (\texttt{\^{}2}).} Runs of two
        or more consecutive capitalized words (excluding stopwords and broken by
        punctuation) are added as \texttt{\^{}2} phrase queries.
        \emph{Pierre Cauchon}, \emph{El Tahrir}, \emph{Turkish Republic} are
        caught; standalone words like \emph{Italian} or \emph{Adoration} are
        not --- boosting them blindly was tried and \emph{regressed} Ben Affleck
        retrieval to \emph{Helen Hunt} and \emph{Oscar Wilde}.

  \item \textbf{Three-bucket rerank} of the top-50 hits, in priority order:
        \begin{itemize}
          \item \emph{Demote meta-pages} (\texttt{List of \ldots},
                \texttt{Index of \ldots}, \texttt{Timeline of \ldots},
                \texttt{(disambiguation)}). They contain many topical terms but
                are never the Jeopardy answer.
          \item \emph{Demote topical decoys}: pages whose title's word set is a
                strict subset of the clue's word set. Drops \emph{Nile} when
                the answer is \emph{Cairo}; almost never fires on the real
                answer because Jeopardy clues do not literally state the answer.
          \item \emph{Promote contiguous-phrase matches}: if the page's title
                appears as a contiguous substring of the clue, its score is
                bumped by $+1000$. Catches the ``answer is the quoted phrase in
                the clue'' cases like \emph{Father Figure (song)}.
        \end{itemize}
\end{enumerate}

\subsection{What Was Tried and Rejected}

\begin{itemize}
  \item \textbf{Appending category to the BM25 query.} Hurt P@1 by 8 points
        ($12/100$). Short, hyped strings like \emph{``1920s NEWS FLASH!''}
        outweigh the clue. The category remains parsed and is wired into
        \texttt{search()}'s signature so it can be re-introduced for
        rerank-only use later.
  \item \textbf{\texttt{\^{}2} boost on every mid-clue capitalized word.} Was
        the v1 improvement; net $15/100$. Words like \emph{Hunting} and
        \emph{Oscar} are capitalized but not the entity Jeopardy is asking
        about.
  \item \textbf{Whoosh BM25F default $B = 0.75$.} See the sweep table in
        Section~3 --- replaced by $B = 0.1$ in both modes.
\end{itemize}

Net effect of mode \texttt{improved}: \textbf{+5 P@1, +0.04 MRR over baseline.}

\subsection{Mode \texttt{llm} --- Claude Reranker over the Top-10}

\texttt{watson.py --mode llm} runs the same \texttt{improved} retrieval
as above and then hands the top-10 candidate titles to
\textbf{Claude Sonnet 4.6} (via the official Anthropic Python SDK),
asking it to pick the single best answer. The picked candidate is
moved to rank~1; the rest keep their relative order. See
\texttt{src/llm\_rerank.py} for the prompt and the disk-cache layer.

\paragraph{Why title-only.} The Whoosh index does not store page
bodies (\texttt{body=stored=False} in \texttt{build\_index.py}), so the
reranker prompt contains only \texttt{(clue, category, list of 10
titles)}. Adding snippets would require either re-streaming the wiki
dump on every query (slow) or rebuilding the index with bodies stored
($\sim$20 min). Since Claude already has the entire Wikipedia subset in
its training data, the title-only prompt turned out to be enough --- the
reranker correctly picks the gold page on every clue where it is
present in the top-10.

\paragraph{Why Sonnet, not Opus.} The task is a constrained
classification (pick 1 of 10 titles), not open-ended reasoning. Sonnet
4.6 is fully adequate and approximately $3\times$ cheaper than Opus 4.7
for this workload, which matters for the $\sim$100-call evaluation
cycle and any re-runs during prompt iteration.

\paragraph{Disk cache.} Each \texttt{(clue, sorted candidate titles)}
tuple is SHA-256-hashed and the picked index is cached to
\texttt{results/llm\_rerank\_cache.json}. Re-running the evaluation
does not re-bill the API. Sorting the title list before hashing means
tiny BM25 score differences (which can shuffle ties) do not invalidate
the cache.

\paragraph{Result.} P@1 jumps from \textbf{0.25 (improved) to 0.55
(llm)} --- a 30-point absolute improvement, and the largest single
gain in the project. Crucially,
$\text{P@1} = \text{P@5} = \text{P@10} = \text{MRR} = 0.55$: every
correct answer that the IR system surfaced in the top-10 was correctly
identified by the reranker. The LLM closes the rank-quality gap
entirely; the remaining errors are pure retrieval failures (the gold
page is not in the top-10 at all). See Section~\ref{sec:error} for
the bucketed breakdown of those residual misses.

\paragraph{What the LLM gets right that BM25 cannot.} Three classes of
clue that Section~\ref{sec:error} flagged as unreachable for lexical
retrieval are now solved:

\begin{itemize}
  \item \emph{Inferential}: \emph{``For the brief time he attended, he
        was a rebel with a cause, even landing a lead role in a 1950
        stage production''} $\to$ \emph{James Dean} (gold). The
        lexical retriever needed a bridge through the
        \emph{Rebel Without a Cause} film article; Claude makes it
        directly.
  \item \emph{Pun categories}: clues under \texttt{"TIN" MEN}
        (\emph{Tintoretto}), \texttt{COMPLETE DOM-INATION}
        (\emph{William the Conqueror}), etc.\ now resolve because the
        reranker decodes the pun.
  \item \emph{Indirect lookups}: \emph{``The Naples Museum of Art is
        in this state's Collier County''} $\to$ \emph{Florida} (gold),
        not the museum. The reranker steps from the museum article to
        its location.
\end{itemize}

\paragraph{The one regression.} \emph{Ottoman Empire} drops to
off-list because the \texttt{improved} retrieval does not surface it
in the top-10 --- the capitalized run \emph{Turkish Republic} in the
clue boosts the wrong cluster of pages and pushes \emph{Ottoman Empire}
out. The reranker can only choose from what retrieval returns; it
cannot recover answers the IR layer never saw.

\section{Notes on Reproducibility}

\begin{itemize}
  \item Whoosh stores the index in \texttt{index/}; re-running
        \texttt{build\_index.py} wipes and rebuilds. Use \texttt{--keep} to add
        to an existing index.
  \item The full index occupies a few GB on disk and takes approximately
        30--60 minutes to build on a laptop. Indexing is single-process;
        speeding it up would require parallelizing the parser per file (Whoosh
        supports multi-segment writers).
  \item \texttt{data/questions.txt} is the only source of evaluation; no
        validation split is held out (100 questions is too small to split
        meaningfully).
  \item The \texttt{llm} mode's API decisions are cached in
        \texttt{results/llm\_rerank\_cache.json} (committed). Re-running
        \texttt{python src/compare\_modes.py --modes baseline improved llm}
        reproduces the reported numbers exactly without making any new
        API calls; deleting the cache file forces a fresh round of
        Claude calls (cost: $\sim$\$0.10--\$0.50 on Sonnet 4.6 for the
        full 100 questions).
\end{itemize}
